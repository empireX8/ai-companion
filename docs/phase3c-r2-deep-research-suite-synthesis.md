## 1. Executive synthesis

### Core conclusion

```text
The Deep Research is valuable for mechanics, but not for category direction.
MindLab should not become a better Rosebud, Mindsera, Stoic, Finch, or Character.AI.
The suite direction overrides competitor category gravity.
```

The Deep Research correctly identifies retention mechanics, trust patterns, and competitive density in AI journaling / self-reflection. It incorrectly treats MindLab as primarily a **mobile consumer habit product** competing in the Rosebud / Mindsera / Stoic lane. That framing would pull the product toward capture → chat → insight loops without inspectable model objects, evidence receipts, or a web workbench — eroding MindOS differentiation.

### Correct product interpretation

```text
MindOS is the central intelligence layer.
The suite is the meta-product.
Web is the operating console.
Mobile is the daily field instrument.
Competitor mechanics should be extracted and reassigned into that system.
```

**Decision rule for R2 and all future slices:**

| Question | Answer lens |
|----------|-------------|
| What do we borrow? | Mechanics that strengthen evidence-backed operating intelligence |
| What do we reject? | Category identity, companion framing, mood-first UX, chatbot shell |
| Where does it belong? | Suite / web / mobile / onboarding / pricing / agents / later |
| Does it define MindOS? | Only if it strengthens mirror objects + receipts + model movement |

## 2. Research value vs research risk

### What the research usefully confirms

| Finding | Suite implication |
|---------|-------------------|
| AI journaling / self-care / wellness space is crowded | MindOS must **not** compete on journal-app category terms |
| Rosebud, Mindsera, Stoic already cover capture → reflection → insight | Differentiation = **inspectable evidence-linked self-model**, not better prompts |
| Competitors do not strongly own inspectable evidence-linked self-models | User Map, receipts, model updates, Active Questions remain defensible |
| Receipts, model updates, active questions, Watch For, fieldwork remain differentiated | Double down on object model; do not collapse into chat summaries |
| Daily return loops matter | Mobile Today mirror + notifications — **already partially implemented (3C-2)** |
| Weekly synthesis matters | What Changed / model movement as **major retention loop** — web-primary, mobile-aware |
| Resurfacing matters | Watch For + Timeline + Readwise-style review — not random push notifications |
| Trust / export / delete / privacy matter | Non-negotiable suite trust architecture; never paywall export/delete |
| Chat-first and companion-first approaches carry category risk | Chat = interpretation layer; not home shell |
| Action-linked progress is powerful if adapted correctly | Fieldwork/actions change model state — not streaks or pet care |
| Web / workbench depth matters for structured systems | Web is long-term console — not a responsive mirror of mobile |

### What the research risks over-weighting

| Risk | Why it is dangerous for MindOS |
|------|-------------------------------|
| Mobile-first consumer habit app thinking | Collapses suite into App Store journal competitor |
| Journaling-first product architecture | Makes chat/capture the centre instead of mirror objects |
| Wellness / self-care category gravity | Therapy-coded trust language and mood-dashboard heroes |
| Therapy-adjacent trust language | Violates product truth; blocks FinanceOS / StudyOS extension |
| Dark / private / intimate aesthetic as brand centre | Conflicts with light-first multi-domain suite (R1 locked) |
| Companion / chatbot engagement | Parasocial retention replaces evidence-backed understanding |
| Mood / streak / badge progress logic | Fake progress without model movement |
| Psychometric / personality dashboards as product centre | Static labels replace evidence-backed User Map |

**R2 rule:** When research recommends a mechanic, ask: *Does this make MindOS a better operating system for understanding yourself, or a better journal/companion app?* If the latter, reject or reassign to a narrow mobile/onboarding sub-mechanic only.

## 5. Destination buckets

All useful research mechanics sorted into six implementation-planning buckets. **None of these buckets authorize implementation in R2.**

### Whole OS suite

- Evidence object model (receipts, events, links)
- Shared User Map / MindOS central model
- Cross-domain evidence links (future)
- What Changed / Model Updates
- Trust architecture (export, delete, privacy, basis-on-tap)
- Receipts as proof layer
- Goals / decisions (as model objects mature)
- Fieldwork feedback → model movement
- Watch For resurfacing logic
- Model update generation and publish policy (existing backend)

### Web operating console

- Full User Map list, detail, edit/curate (when allowed)
- Evidence / receipt archive and advanced search
- Domain dashboards (FinanceOS, StudyOS, etc. — future)
- Model editing and diffing
- Investigations / Active Questions depth
- Long-form synthesis and export
- Imports / connectors (ChatGPT import exists; extend later)
- Future agent control plane
- What Changed full feed and evidence expand
- Cross-domain model view

### Mobile daily field instrument

- Quick capture (text, voice, photo) — **partially shipped**
- Today living mirror — **shipped (3C-2)**
- Check-ins without mood-first framing
- Watch For prompts (max 1 preview on Today)
- Lightweight Explore
- Fieldwork / action completion
- Notifications / resurfacing (future)
- Fast evidence metadata review — **basis summaries shipped (3C-SAFE)**
- Session continuity (Journal Chat / Explore)

### Onboarding

- First mirror object (evidence-backed, not fake insight)
- Demo state with sample evidence structure (not fabricated claims)
- “Capture something you want to understand better”
- No therapy survey; no mood onboarding
- No full profile setup first
- No blank chatbot homepage first
- Immediate evidence-backed payoff within first session

### Pricing / premium

- Free activation layer (capture + basic mirror)
- Paid depth / scale / workbench / connectors
- Premium web operating console value
- **Never** paywall export or delete
- Possible suite / domain OS pricing later
- Human expert review layer — **deferred**

### Future agents / domain OSs

- FinanceOS / StudyOS / SpeakOS / CreatorOS product architecture (R9)
- Domain operating agents — not characters
- Agent tasks tied to user permission, evidence, goals, workflows
- MindOS agent for cross-domain synthesis
- Wrtn-style tool breadth **only** inside controlled agent/control layer (R10)

## 6. Product decisions locked by R2

These decisions are **additive to R1** and prepare R3–R10. They do not authorize implementation.

1. **MindOS is not an AI journal.**
2. **MindOS is not therapy / wellness software.**
3. **Web is the long-term full operating console.**
4. **Mobile is the daily field instrument.**
5. **Light-first suite identity remains primary.**
6. **Dark graphite is deferred** as optional focus mode — not brand centre.
7. **Chat is not the home shell.** Pages and mirror objects activate; chat interprets.
8. **Evidence-backed mirror objects are the activation unit** — not chat threads or mood cards.
9. **Receipts are a core trust surface** — basis-on-tap, archive on web, metadata on mobile.
10. **What Changed should become a major retention loop** — web-primary weekly synthesis; mobile-aware hero/feed.
11. **Watch For is the resurfacing mechanic** — Readwise-adapted, not notification spam.
12. **Fieldwork / actions are the progress mechanic** — Finch-adapted model state change, not streaks.
13. **User Map is the cross-domain central model** — not a static profile or personality dashboard.
14. **Domain OSs should feed MindOS later** — evidence in, model updates out; no premature multi-app launch.
15. **Agents belong later as operating agents** — not characters, companions, or therapists (R10).
16. **Pricing should monetize depth, scale, connectors, web workbench, and suite value** — not basic understanding or export/delete.
17. **Human expert layer remains deferred.**
